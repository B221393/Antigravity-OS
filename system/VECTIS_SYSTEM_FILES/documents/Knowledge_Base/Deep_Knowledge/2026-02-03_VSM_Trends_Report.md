# Edge AI & Vision-Language Model Trends Report (Feb 2026)

## 1. Executive Summary: "汎用"から"特化"への揺り戻し

2025年後半から続くトレンドとして、巨大な汎用VLM（Vision-Language Models）の進化と並行して、**「特定のユーティリティタスクに特化した1B以下の超軽量モデル」**の需要が爆発的に増加しています。

特に **GLM-OCR (0.9B)** の登場は、RAG（検索拡張生成）パイプラインにおける**「前処理（Ingestion）コストの削減」**と**「オンデバイス・プライバシー」**への強い要請への回答と言えます。

## 2. Key Players (主要モデル)

### a. DeepSeek-OCR (3B) - "Context Optical Compression"

- **概要**: DeepSeekが2025年末にリリースしたOCR特化モデル。
- **差別化**: 単なるテキスト抽出ではなく、**「視覚情報の圧縮表現（Visual Tokens）」**を直接生成し、後段のLLMに渡すことを想定したアーキテクチャを採用。「人間が読むためのテキスト」ではなく「AIが読むための埋め込み」を出力するモードを持つ点が革新的。
- **GLMとの比較**: 精度はDeepSeekが上だが、3Bというサイズはエッジ（スマホやRaspberry Piなど）での常時稼働には重い。0.9BのGLMはこの隙間を突いている。

### b. Qwen3-VL-2B & Nano - "The All-Rounder"

- **概要**: Alibaba (Qwen) の最新シリーズ。特に2B以下のクラス（Nano）の最適化が著しい。
- **特徴**: OCR特化ではなく汎用VLMだが、ドキュメント理解能力が前世代（Qwen2.5-VL）から飛躍的に向上。動画入力にも対応。
- **ユースケース**: 監視カメラ映像からのテキスト抽出など、動画を含むマルチモーダル処理ではGLM-OCRより有利。

### c. Gemma 3 (Google) - "Mobile First"

- **概要**: Androidデバイス上のNPU（Neural Processing Unit）での動作に特化したモデル群。
- **特徴**: 端末内での推論効率が異常に高く、Pixelシリーズなどのスマホアプリに組み込むなら現状の最適解。GLM-OCR同様、ローカル動作を前提としている。

### d. Chandra OCR (9B) & OlmOCR-2 (8B)

- **概要**: より大規模な「論理推論型OCR」。
- **特徴**: 単に文字を読むだけでなく、「この請求書の合計金額が計算的に合っているか？」といった検証まで行える。
- **立ち位置**: サーバーサイドで動かす高精度モデル。GLM-OCRとは競合せず、むしろGLMで一次処理したデータをこれらで検証するような使い分けがなされている。

## 3. 技術的考察と批判的視点 (Critical Analysis)

### なぜ今「OCR特化」なのか？ (Why Now?)

汎用モデル（GPT-5クラスやGemini 2.5 Proなど）になんでも投げれば解決する時代に、なぜ0.9BのOCRモデルが必要なのか。

1. **Token Economics (トークン経済性)**:
    高精細な画像をそのまま巨大モデルに入力すると、数千〜数万トークンを消費する。GLM-OCRで一度Markdownテキスト（数百トークン）に変換してから巨大モデルに投げる方が、APIコストもVRAMも圧倒的に安上がり。

2. **Privacy & Latency**:
    請求書や医療記録などの機密情報をクラウドに上げずに処理したい需要。0.9Bなら、一般家庭のPCやスマホのバックグラウンドプロセスとして常駐可能。

### 懸念点 (Concerns)

- **幻覚 (Hallucination) の質**: 軽量モデルは、画像が不鮮明な場合に「もっともらしい嘘の数値」を自信満々に出力する傾向がある。DeepSeek-OCRのような「確信度スコア」を出す仕組みがGLM-OCRでどこまで実装されているかが実運用の鍵となる。
- **多言語対応の限界**: 0.9Bの容量で、英語・中国語以外（日本語の縦書きや特殊フォントなど）にどこまでリソースを割けているかは検証が必要。

## 4. 推奨パイプライン構成 (Recommended Pipeline)

現状のローカルLLM環境における「最強の布陣」は以下の通りです。

| Role | Model | Specs | Task |
| :--- | :--- | :--- | :--- |
| **Eye (視覚)** | **GLM-OCR** | 0.9B | 画像 $\to$ Markdown/JSON変換 (常駐) |
| **Brain (思考)** | **DeepSeek-R1** / Llama 4 | 7B - 32B | データの要約、構造化、コード生成 |
| **Check (検証)** | **Qwen2.5-Math** | 7B | 数値計算の整合性チェック (必要な場合のみ) |
